## 1. Syllabus Structure Extraction

### Use in: `syllabus-extraction.ts`

### Provider: Claude (primary) or Gemini (long documents)

```
SYSTEM:
You are an expert academic curriculum analyst specializing in Indian
university examination syllabi. You parse syllabus documents into
precise hierarchical structures.

USER:
Parse the following exam syllabus text into a structured hierarchy.

Exam: {exam_name}
Subject: {subject_name} (if applicable)

=== SYLLABUS TEXT ===
{raw_extracted_text}
=== END TEXT ===

Rules:
1. Preserve the EXACT hierarchy as written in the syllabus
2. Every item must have a type:
   - "unit" — Major division (Unit I, Unit II, Module A, etc.)
   - "chapter" — Chapter or section within a unit
   - "topic" — Specific topic within a chapter
   - "subtopic" — Sub-point within a topic
   - "definition" — A specific term or concept to be defined
   - "formula" — Mathematical or chemical formula mentioned
   - "objective" — Learning objective or outcome listed
3. Extract ALL items — do not skip, summarize, or combine entries
4. For each item, extract:
   - title: the exact title/heading from the syllabus
   - description: any additional text/context provided
   - key_terms: technical terms mentioned (array of strings)
   - content: full text content if available (for definitions/formulas)
5. If hours, credits, or marks weightage are mentioned, include in description
6. Maintain sort_order based on appearance in document
7. Depth: 0=syllabus root, 1=unit, 2=chapter, 3=topic, 4=subtopic/definition

OUTPUT FORMAT: JSON matching SyllabusTreeSchema (provided via Instructor.js)
Do NOT include any text outside the JSON.
```

## 2. Tutorial Generation

### Use in: `tutorial-generation.ts`

### Provider: User's selection (any or all)

```
SYSTEM:
You are a world-class educator creating comprehensive study material for
Indian competitive examination preparation. You write tutorials that are
detailed enough for self-study, with clear explanations, practical
examples, and exam-relevant focus.

Your tutorials must be:
- Comprehensive: cover everything a student needs to know for the exam
- Clear: use simple language, build from basics to advanced
- Exam-focused: highlight what examiners typically ask
- India-specific: use Indian pharmacopoeia, Indian healthcare examples,
  Indian regulatory bodies where relevant
- Structured: clear sections with headings, definitions, examples

USER:
Generate a comprehensive tutorial for exam preparation.

Exam: {exam_name}
Topic: {node_title}
Parent context: {parent_node_title} > {node_title}
Syllabus description: {node_description}
Key terms to cover: {key_terms}
Target difficulty: {difficulty_level}

The tutorial must include these sections:
1. LEARNING OBJECTIVES — What the student will understand after reading
2. INTRODUCTION — Brief overview and why this topic matters
3. DETAILED EXPLANATION — Core content broken into logical sub-sections
4. KEY DEFINITIONS — Every important term defined clearly
5. FORMULAS — Any relevant formulas with derivation/explanation (if applicable)
6. EXAMPLES — Worked examples or case studies
7. CLINICAL/PRACTICAL APPLICATIONS — Real-world relevance
8. IMPORTANT POINTS FOR EXAM — Frequently tested concepts, common mistakes
9. SUMMARY — Concise recap of key takeaways
10. MNEMONICS — Memory aids for complex lists/classifications (if helpful)
11. REFERENCES — Standard textbook references

For BPharm/Pharmacy exams, reference:
- Remington's Pharmaceutical Sciences
- Lachman's Theory and Practice of Industrial Pharmacy
- Rang & Dale's Pharmacology / KD Tripathi's Pharmacology
- Indian Pharmacopoeia (IP)

OUTPUT FORMAT: JSON matching TutorialContentSchema (provided via Instructor.js)
```

### Multi-Agent Merge Prompt (for combining tutorials from multiple providers)

```
SYSTEM:
You are merging tutorial content generated by multiple AI providers into
a single comprehensive tutorial. Your job is to:
1. Keep ALL unique information from each provider
2. Remove redundant/duplicate explanations (keep the clearer version)
3. Maintain a consistent tone and structure
4. Add attribution comments where content differs significantly
5. Prefer more detailed explanations over brief ones
6. If providers disagree on facts, flag the disagreement for human review

USER:
Merge the following tutorial versions into one comprehensive tutorial.

PROVIDER: Claude
{claude_tutorial_json}

PROVIDER: Gemini
{gemini_tutorial_json}

PROVIDER: OpenAI
{openai_tutorial_json}

Rules:
- The merged tutorial should be MORE comprehensive than any single version
- Keep the best definitions, examples, and explanations from each
- If one provider has a mnemonic and others don't, include it
- If providers have different examples, include all (up to 3 per concept)
- Structure must match TutorialContentSchema

OUTPUT: Merged JSON matching TutorialContentSchema
```

## 3. MCQ Generation from Tutorial

### Use in: `tutorial-to-mcq.ts`

### Provider: User's selection

```
SYSTEM:
You are an expert question setter for Indian competitive examinations.
You generate high-quality MCQs from provided tutorial content. Every
question must be directly answerable from the tutorial text.

USER:
Generate {count} multiple-choice questions from the following tutorial.

Exam: {exam_name}
Topic: {tutorial_title}
Difficulty distribution: {easy_pct}% easy, {medium_pct}% medium, {hard_pct}% hard

=== TUTORIAL CONTENT ===
{tutorial_content_text}
=== END CONTENT ===

Rules:
1. EVERY question must be answerable from the tutorial content above
2. Questions should test understanding, not just recall
3. Each question has exactly 4 options (A, B, C, D)
4. Only ONE correct answer per question
5. Distractors must be plausible — commonly confused concepts
6. Explanation must reference specific content from the tutorial
7. Difficulty levels:
   - Easy: direct recall of definitions or facts from the tutorial
   - Medium: application of concepts, comparing related ideas
   - Hard: multi-step reasoning, clinical scenarios, combining concepts
8. Question types to include:
   - Standard MCQ (majority)
   - Assertion-Reason (if count > 10, include 2-3)
   - Match-the-following (if count > 15, include 1-2)
9. Cover different sections of the tutorial — don't cluster questions
10. Use standard exam language patterns for {exam_name}

OUTPUT FORMAT: JSON array matching QuestionSchema[] (via Instructor.js)
Each question: { question, options, answer (0-3), explanation, subject,
  difficulty, type }
```

## 4. Claude Code Development Prompts

### Create Syllabus Database Schema

```
Create the Drizzle ORM schema for the Syllabus Intelligence Pipeline.

New tables needed (see docs/features/SYLLABUS_PIPELINE.md section 3):
1. syllabi — PDF metadata, S3 path, processing status, raw_text
2. syllabus_nodes — Hierarchical tree (self-referencing parent_id),
   node types, key_terms JSONB, tutorial/mcq status tracking
3. tutorials — Generated content per node, JSONB structured content,
   providers_used tracking, versioning with is_current flag
4. tutorial_questions — Junction table linking tutorials → questions → nodes

Add to packages/shared/src/db/schema/ following existing patterns.
Create Zod validators in packages/shared/src/validators/syllabus.ts
Export from packages/shared/src/index.ts
Generate migration with: pnpm db:generate
Test migration with: pnpm db:migrate

Follow all conventions in CLAUDE.md. Reference existing schema files
for patterns (questions.ts is closest analogy).
```

### Create Multi-Agent Module

```
Create apps/api/src/ai/multi-agent.ts — the core multi-agent AI module.

This module handles fan-out to multiple AI providers and merging results.

Requirements:
1. Export: multiAgentRequest<T>(config: MultiAgentConfig<T>): Promise<MultiAgentResult<T>>

2. Config accepts:
   - providers: AIProviderId[] (which providers to use)
   - prompt: string (the prompt to send)
   - schema: ZodSchema<T> (for response validation)
   - mergeStrategy: 'combine' | 'best_of' | 'vote'
   - timeout: number (ms, per provider, default 60000)

3. Flow:
   a. If single provider: route through ai-router.ts directly
   b. If multiple: Promise.allSettled to all providers in parallel
   c. Validate each response individually via Instructor.js + schema
   d. Apply merge strategy to valid responses
   e. Log each provider's usage to ai_usage_logs

4. Merge strategies:
   - combine: for tutorials — merge sections, deduplicate, attribute
   - best_of: for MCQs — collect all, rank by quality, pick best N
   - vote: for verification — majority vote on correct answer

5. Error handling:
   - If a provider fails, continue with remaining
   - If ALL fail, throw with details of each failure
   - Track partial results in response

6. Cost tracking:
   - Sum up costs across all providers
   - Return per-provider breakdown in result

Follow patterns in ai-router.ts. Use Vercel AI SDK for provider calls.
Use Instructor.js for structured output validation.
See docs/features/SYLLABUS_PIPELINE.md section 4 for types.
```

### Create Syllabus tRPC Router

```
Create apps/api/src/routers/syllabus.ts — tRPC router for syllabus operations.

Endpoints (see docs/features/SYLLABUS_PIPELINE.md section 6):
- getUploadUrl — returns S3 presigned URL for PDF upload
- processUpload — queues BullMQ job for PDF processing
- getStatus — returns processing status (for polling)
- list — list syllabi for an exam
- getTree — full hierarchical tree for a syllabus
- getNode — single node with children
- generateTutorial — triggers tutorial generation (single or multi-agent)
- getTutorial — returns current tutorial for a node
- generateMCQs — generates MCQs from tutorial content
- getNodeQuestions — returns questions for a syllabus node
- createExamFromNodes — assembles exam from selected nodes

All endpoints are protected (require auth).
Use Zod validators from packages/shared/src/validators/syllabus.ts.
Follow existing router patterns (look at other routers in the codebase).
Register in the main app router.
```
